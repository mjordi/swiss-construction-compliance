## Proposal S — Normalize linked-case checklist state after protocol finalization

- **Target user and job-to-be-done:** Swiss construction owners, contractors, and project admins finalizing a digital acceptance protocol that is linked to an existing defect case.
- **Problem:** The dashboard protocol finalization flow auto-marks `defectDocumented`, but it writes the linked case checklist by spreading the raw persisted JSON. If the persisted checklist is `null` or sparse, the payload can omit sibling readiness keys. Downstream case/Vault readiness views normalize at render time today, but persistence remains inconsistent and weakens the audit contract for future cross-workflow activity views.
- **Scope:** In `app/dashboard/page.tsx`, import and use the shared `normalizeFollowUpChecklistState` helper when marking linked-case `defectDocumented`, layering persisted checklist data over the selected case timeline defaults before updating. Update focused dashboard linked-case tests to assert a complete normalized checklist payload for null and sparse persisted checklist responses.
- **Expected impact:** More reliable linked protocol → case checklist synchronization and a stronger prerequisite for cross-workflow audit summaries.
- **User-visible outcome:** After a linked protocol is finalized, the Cases/Vault readiness state remains complete and consistent instead of depending on sparse persisted JSON.
- **Competitive / best-product rationale:** A legal compliance product should keep evidence-readiness state canonical at workflow boundaries, not only visually normalize it later.
- **Risk:** 2/10 — uses an existing shared helper and focused tests; no schema or UI layout change.
- **Effort/size:** Small (S).
- **Phase-1 slice:** This proposal is already the bounded phase-1 slice.

## Proposal M — Add a read-only linked evidence summary to the protocol final-review step

- **Target user and job-to-be-done:** A site manager or project admin reviewing a linked protocol before finalizing the PDF and saving legal evidence.
- **Problem:** The final-review step shows linked case status/deadline context but not the current evidence readiness/checklist state, so the user may finalize without realizing the case still lacks attached evidence or reminder export.
- **Scope:** Add a compact read-only final-review checklist/evidence summary from the selected case context and persisted checklist state. Keep it informational only; do not change PDF schema, persistence, deadline math, or checklist toggles.
- **Expected impact:** Better review confidence before finalizing legally relevant protocol evidence.
- **User-visible outcome:** The final-review screen names the linked case's readiness gaps before the user commits the protocol.
- **Competitive / best-product rationale:** Best-in-market tools make audit gaps visible at the decision point, not only after saving.
- **Risk:** 5/10 — needs careful copy, async selected-case state handling, and tests around pending/missing linked case context.
- **Effort/size:** Medium (M).
- **Phase-1 slice:** Show only normalized completed/total checklist count plus evidence/reminder labels, with no mutation controls.

## Proposal L — Build a unified audit-ready activity timeline across cases, protocols, Vault evidence, and reminders

- **Target user and job-to-be-done:** Swiss construction teams preparing a dispute-ready audit trail for a project.
- **Problem:** Recent slices expose individual artifact signals, but users still lack a single chronological audit view spanning defect discovery, protocol finalization, reminder export, Vault evidence lifecycle, and deadline status.
- **Scope:** Create a derived timeline model and first read-only UI surface joining cases, protocols, Vault project status, checklist readiness, reminders, and deadline context.
- **Expected impact:** Major step-change toward BauCompliance as a legal audit cockpit rather than separate workflow pages.
- **User-visible outcome:** Users can review one project activity trail and understand what is complete, pending, or legally urgent.
- **Competitive / best-product rationale:** A best-in-market construction compliance product should produce an audit narrative, not just isolated task lists.
- **Risk:** 8/10 — larger data/model/copy/test surface and potential overlap with recent case/Vault work.
- **Effort/size:** Large / strategic (L).
- **Phase-1 slice:** First build the normalized read-only timeline helper for a single Cases card, using existing case/protocol counts and checklist state only.
