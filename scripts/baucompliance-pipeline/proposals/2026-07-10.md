## Proposal S — Mark linked-case defect evidence even when the case checklist is empty

- **Target user and job-to-be-done:** Project admin finalizing an acceptance protocol linked to an existing defect case, needing the case readiness ledger to show protocol evidence was captured.
- **Problem:** The dashboard finalization flow currently attempts to mark `defectDocumented` only when the linked case already has a checklist object. Cases with `null` or empty checklist data can keep showing the defect-documentation item incomplete after a protocol is finalized, making the legal readiness ledger under-report captured evidence.
- **Scope:** Normalize the loaded linked-case checklist before updating it, set `defectDocumented: true` for linked cases regardless of whether persisted checklist JSON already exists, and add a focused regression test in `__tests__/dashboard-linked-case-retry.test.tsx`.
- **Expected impact:** Prevents a real audit-readiness mismatch in a core protocol → case workflow.
- **User-visible outcome:** After finalizing a linked protocol, the case checklist can immediately reflect that defect/no-defect protocol evidence is documented, even for newly created cases with no prior checklist state.
- **Competitive / best-product rationale:** Best-in-market compliance software should keep legal artifacts and readiness ledgers synchronized automatically, not require users to discover and manually repair silent state gaps.
- **Risk (1-10):** 2
- **Effort/size:** S
- **Phase-1 slice:** Complete as proposed.

## Proposal M — Add a case-linked protocol activity snapshot on the case row

- **Target user and job-to-be-done:** Contractor reviewing a case row and deciding whether evidence/protocol work is complete enough before a notice deadline.
- **Problem:** Cases show a protocol count but not the most recent protocol status or whether protocol evidence is the source of checklist readiness.
- **Scope:** Add a compact case-row activity snapshot sourced from existing protocol counts and linked-case checklist state, without creating a new activity table.
- **Expected impact:** Makes case readiness more explainable and moves toward a compliance activity cockpit.
- **User-visible outcome:** Users see a clearer signal that protocol evidence exists for the case rather than just a number.
- **Competitive / best-product rationale:** Strong products summarize proof chains at the point of action.
- **Risk (1-10):** 5
- **Effort/size:** M
- **Phase-1 slice:** Read-only display only; no persistence schema changes.

## Proposal L — Build a unified audit-ready activity timeline across cases, protocols, vault evidence, and reminders

- **Target user and job-to-be-done:** Team lead preparing for dispute, warranty review, or internal quality audit who needs one chronological chain of compliance events.
- **Problem:** Evidence is distributed across cases, protocol PDFs, reminder exports, vault project cards, and checklist state. Users must mentally assemble the audit trail.
- **Scope:** Define an activity model and dashboard surface that aggregates protocol finalizations, case reminder exports, evidence/vault updates, and readiness changes.
- **Expected impact:** Step-change market differentiation and lower audit preparation effort.
- **User-visible outcome:** A single timeline answering “what happened, when, and what proof exists?”
- **Competitive / best-product rationale:** An audit-ready timeline would make BauCompliance feel like a legal operations system rather than a collection of forms.
- **Risk (1-10):** 8
- **Effort/size:** L
- **Phase-1 slice:** Start with a read-only timeline from already-confirmed events after artifact-to-checklist synchronization is proven reliable.
