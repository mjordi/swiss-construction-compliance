# Daily Product Improvement Proposals — 2026-06-10

## PM insights

- Recent merged work has concentrated on small, high-confidence reliability and UX fixes around dashboard/case/deadline workflows, usually with focused Vitest coverage and pipeline artifacts.
- The current settings page still renders profile/password labels as visual-only `<label>` elements without `htmlFor`/`id` wiring, unlike the product goal of reliable, accessible dashboard workflows. This is a narrow accessibility and testability gap with low implementation risk.
- Larger opportunities remain around consistent URL-state helpers and cross-dashboard legal deadline guidance, but those touch broader product surfaces and are less suitable for a one-day autonomous PR while recent throughput is mostly S-sized patches.

## Proposal S — Wire settings form labels to their inputs

- **Problem:** The settings page shows labels for name, company, and new password fields, but they are not programmatically associated with the inputs. Screen-reader users and label-based tests cannot reliably target these controls.
- **Scope:** Add stable `id` values and `htmlFor` wiring for the three settings form labels. Add focused tests that verify the inputs are discoverable by accessible label in the existing settings test coverage.
- **Expected impact:** Improves dashboard accessibility with minimal risk; makes future settings tests less dependent on placeholders/display values.
- **Risk:** 1/10
- **Effort/size:** S

## Proposal M — Extract shared query-param sanitization helpers for dashboard list pages

- **Problem:** Cases, vault, and deadline pages each manage URL search params with local parsing, cleanup, and stale feedback invalidation. The patterns are similar enough that future fixes risk diverging.
- **Scope:** Introduce a small shared helper for independent param hydration/cleanup and migrate one or two dashboard list pages with regression tests for preserving valid sibling params.
- **Expected impact:** Reduces recurring URL-state regressions and makes future share-link work safer.
- **Risk:** 4/10
- **Effort/size:** M

## Proposal L — Build a legal deadline guidance layer across dashboard cases

- **Problem:** Dashboard cases expose deadline statuses, but users do not yet get a strategic, step-by-step Swiss construction law guidance layer explaining next actions by legal regime and status.
- **Scope:** Add a guidance model that maps case timeline/regime/status/checklist progress to recommended actions, then surface it on cases, vault, and protocol handoff screens.
- **Expected impact:** Strong domain value and differentiation for BauCompliance, moving from tracking toward guided compliance action.
- **Risk:** 7/10
- **Effort/size:** L
