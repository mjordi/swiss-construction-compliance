## Proposal S — Associate Ruegefrist calculator visible date labels with their date inputs

- **Target user and job-to-be-done:** Swiss contractors/owners using the public Ruegefrist calculator to enter contract and defect-discovery dates accurately, including keyboard/assistive-tech users and users who click visible labels before entering dates.
- **Problem:** The date inputs have accessible names via `aria-label`, but the visible labels are not programmatically associated with the inputs. Clicking the visible label does not target the field, and the UI contract is weaker than best-in-market legal tools where visible legal terms directly label the exact input being submitted/exported.
- **Scope:** Add stable IDs to the contract/discovery date inputs, wire the visible labels with `htmlFor`, and add a focused regression test asserting the visible labels control their corresponding inputs.
- **Expected impact:** Small but meaningful legal-input trust/accessibility improvement on a public conversion/activation surface.
- **User-visible outcome:** Users can click/tap the visible contract/discovery labels to focus the correct date input; assistive technologies get a direct visible-label relationship.
- **Competitive / best-product rationale:** A best-in-market Swiss legal-compliance calculator should make legally sensitive date fields unambiguous and accessible without relying on invisible-only labeling.
- **Risk (1-10):** 2
- **Effort/size:** S
- **Phase-1 slice:** Complete in one PR with one component edit and focused test coverage.

## Proposal M — Public-to-dashboard case handoff from calculated Ruegefrist result

- **Target user and job-to-be-done:** A project lead who calculates a 60-day notice deadline and then wants to continue into a tracked compliance case with the same dates.
- **Problem:** The public calculator produces a legally useful result and calendar export, but the next action toward a persistent case workflow is not yet explicit.
- **Scope:** Add a dashboard handoff CTA after calculation that preserves contract/discovery dates in query params for a future/new case flow, reusing existing query-state and draft-loading patterns where possible.
- **Expected impact:** Higher activation and cross-workflow continuity from public legal calculation to dashboard case tracking.
- **User-visible outcome:** After calculating, the user sees a clear next step to track the deadline/case rather than only downloading an ICS file.
- **Competitive / best-product rationale:** A market-leading tool should bridge one-off legal calculations into tracked audit-ready workflows.
- **Risk (1-10):** 5
- **Effort/size:** M
- **Phase-1 slice:** Add CTA and prefilled URL contract only if an existing dashboard destination can consume it safely; otherwise first add a pure query contract/test in the destination.

## Proposal L — Case activity timeline across protocols, vault evidence, checklist, reminders, and finalization

- **Target user and job-to-be-done:** Project teams and legal/admin reviewers reconstructing what happened for a defect case and whether the case is audit-ready.
- **Problem:** Case state exists across checklist progress, vault evidence, handover protocols, reminders, and finalization flows, but there is no unified chronological activity view.
- **Scope:** Create an audit timeline model and eventually surface it in cases/dashboard/vault contexts.
- **Expected impact:** Strategic legal trust improvement: users can understand and prove the sequence of compliance actions.
- **User-visible outcome:** A single timeline answers when evidence was attached, notice deadlines were calculated/exported, protocol steps happened, and checklist items changed.
- **Competitive / best-product rationale:** Best-in-market compliance systems make audit readiness obvious, not scattered across screens.
- **Risk (1-10):** 7
- **Effort/size:** L / strategic
- **Phase-1 slice:** Add a pure timeline derivation helper with fixtures before any UI/persistence change.
