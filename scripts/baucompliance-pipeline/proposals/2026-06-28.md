## Proposal S — Add dashboard-case handoff from Ruegefrist results

- **Target user and job-to-be-done:** Contractors/project admins who calculate a Ruegefrist deadline and want to track it as a compliance case.
- **Problem:** After calculating the deadline, the user can share/export reminders but must manually navigate to cases and re-enter the contract/discovery dates, risking copy errors and drop-off.
- **Scope:** Add a result CTA from `components/ruegefrist-calculator.tsx` to `/dashboard/cases?contract=YYYY-MM-DD&discovery=YYYY-MM-DD`; teach `app/dashboard/cases/page.tsx` to consume valid `contract`/`discovery` params, open the create-case form, prefill those dates, and remove the consumed params while preserving other filters/search params.
- **Expected impact:** Reduces duplicate entry and turns a public legal calculation into an actionable tracked workflow.
- **User-visible outcome:** After calculating, users see a case-tracking CTA; clicking it opens the cases dashboard with dates already filled in.
- **Competitive / best-product rationale:** Best compliance tools bridge calculators into operational records instead of leaving legal deadlines as one-off outputs.
- **Risk:** 3/10 — touches existing URL-state logic but the slice can be covered by focused tests.
- **Effort/size:** S-to-M.
- **Phase-1 slice:** Date-only handoff; no project-name inference, auth/onboarding funnel, or persisted case creation automation.

## Proposal M — First activity-timeline derivation helper for compliance cases

- **Target user and job-to-be-done:** Project teams reviewing whether each case has notice, evidence, protocol, reminder, and finalization actions completed.
- **Problem:** Case status is split across checklist, protocols, vault handoffs, and dashboard finalization, making audit readiness harder to explain.
- **Scope:** Add a pure helper and fixtures that derive timeline event models from existing case/protocol/checklist inputs; no UI yet.
- **Expected impact:** Provides a safe data-model foundation for a future visible cross-workflow timeline.
- **User-visible outcome:** None in this phase; future UI becomes safer to ship.
- **Competitive / best-product rationale:** Audit-ready timeline views are strategically important, but the pure model phase alone is not user-visible today.
- **Risk:** 5/10 — domain-model choices can affect later UI.
- **Effort/size:** Medium.
- **Phase-1 slice:** Helper + fixtures only.

## Proposal L — Guided public-to-dashboard onboarding flow after legal calculation

- **Target user and job-to-be-done:** Anonymous/public users who discover BauCompliance through the Ruegefrist calculator and need to create a durable compliance workspace.
- **Problem:** The public calculator demonstrates value but does not guide users through login/signup, case creation, evidence upload, and reminder planning as one conversion journey.
- **Scope:** Design and implement a guided post-calculation flow with auth handoff, prefilled case creation, evidence-vault prompt, and reminder export/status tracking.
- **Expected impact:** Higher activation and stronger market positioning as an end-to-end compliance product.
- **User-visible outcome:** A complete “track this deadline” journey from calculation to managed case.
- **Competitive / best-product rationale:** Step-change from tool page to workflow platform.
- **Risk:** 8/10 — crosses auth, dashboard, cases, vault, reminders, and onboarding copy.
- **Effort/size:** Large/strategic.
- **Phase-1 slice:** The S proposal’s date-only handoff is the safest credible first slice.
