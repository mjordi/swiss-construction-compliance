## Small (S) — Associate Ruegefrist calculator visible date labels with inputs

- **Target user and job-to-be-done:** Keyboard and assistive-technology users entering contract and discovery dates in the public Rügefrist calculator.
- **Problem:** The calculator inputs are currently accessible via `aria-label`, but the visible labels are not programmatically connected with `htmlFor`/`id`, leaving a small accessibility consistency gap compared with other recently hardened forms.
- **Scope:** Add stable ids to the two date inputs, connect the visible labels, and adjust the focused calculator accessibility tests to use the visible label association.
- **Expected impact:** Better form semantics and lower assistive-tech friction on a public conversion surface.
- **User-visible outcome:** Users can activate and understand the contract/discovery date fields via their visible labels.
- **Competitive / best-product rationale:** Trustworthy legal calculators should be accessible by default.
- **Risk (1-10):** 2
- **Effort/size:** Small
- **Phase-1 slice:** N/A

## Medium (M) — Canonicalize invalid cases filter query params

- **Target user and job-to-be-done:** Project teams and support users sharing filtered cases dashboard URLs for legal triage review.
- **Problem:** Cases filters parse invalid `regime`, `status`, and `sort` params to safe defaults, but externally introduced invalid params can remain in the URL when the parsed filter state is unchanged. That makes share links look more specific than the UI state actually is.
- **Scope:** Normalize invalid cases filter params whenever search params are consumed, preserve valid sibling params (`q`, `status=triage`) and unrelated query params, and add focused regression tests for initial and external invalid-param cleanup.
- **Expected impact:** More trustworthy shared cases views and fewer support/debugging surprises from ignored legal-triage filters.
- **User-visible outcome:** A cases URL like `?regime=bad&status=triage&sort=random&q=Riverside` becomes a canonical URL that keeps valid triage/search context and removes ignored filters.
- **Competitive / best-product rationale:** Best-in-market compliance tools should keep shared audit/triage links canonical and explainable, especially when filters imply legal status.
- **Risk (1-10):** 3
- **Effort/size:** Medium
- **Phase-1 slice:** Limit the slice to query-param canonicalization for existing filters only; no new filter UI, toasts, or analytics.

## Large / strategic (L) — Case activity timeline across protocols, vault, and reminders

- **Target user and job-to-be-done:** Owners and project managers preparing a defensible audit trail for defect follow-up and handover.
- **Problem:** Evidence uploads, protocols, checklist progress, reminders, and case state are still split across multiple surfaces, making it hard to answer “what happened and what remains legally relevant?”
- **Scope:** Build a case activity timeline spanning protocol creation, vault evidence, checklist actions, deadline reminders, and archive transitions.
- **Expected impact:** Strong strategic differentiation for legal defensibility and project handover clarity.
- **User-visible outcome:** Each case gets a chronological compliance trail that can be reviewed before notices or disputes.
- **Competitive / best-product rationale:** A market-leading Swiss construction-compliance product should turn scattered workflow actions into an audit-ready timeline.
- **Risk (1-10):** 7
- **Effort/size:** Large / strategic
- **Phase-1 slice:** Start with a pure timeline derivation helper and fixture tests using existing case/protocol/vault fields before adding UI or persistence.
