## Small (S) — Stabilize dashboard/settings Supabase client identity

- **Problem:** `app/dashboard/page.tsx` and `app/dashboard/settings/page.tsx` create the Supabase client during render. Effects depend on that client, so fresh-returning factories can trigger avoidable repeated loads or overwrite-prone async paths on rerender.
- **Scope:** Memoize the Supabase client in the dashboard wizard and settings page, mirroring existing cases/vault patterns; add focused regression tests that rerender with a fresh-returning factory and prove the initial loads are not repeated.
- **Expected impact:** Improves reliability for core authenticated workflows and closes the same class of issue recently fixed for cases.
- **Risk (1-10):** 2
- **Effort/size:** S

## Medium (M) — Unify authenticated page Supabase-client stability coverage

- **Scope:** Extract shared test helpers/mocks for authenticated dashboard Supabase loading, add coverage across cases, vault, dashboard wizard, and settings, and document the convention near the Supabase helper.
- **Expected impact:** Stronger long-term regression coverage across the app’s highest-risk async surfaces.
- **Risk (1-10):** 4
- **Effort/size:** M

## Large (L) — Dashboard data-loading architecture phase 1

- **Problem:** Dashboard pages each hand-roll Supabase request lifecycle, stale completion guards, and fallback behavior, creating duplicated async-state risk.
- **Scope:** Design a shared data-loading pattern for authenticated dashboard resources; phase 1 would introduce a small hook for stable client access plus request-id stale completion guards, then migrate one surface.
- **Expected impact:** Strategic reduction in async UI bugs and repeated daily reliability patches.
- **Risk (1-10):** 6
- **Effort/size:** L / strategic
