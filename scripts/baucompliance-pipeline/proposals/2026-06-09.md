## Small (S): Guard password update success against newer edits

- **Problem:** If a user starts a password update and edits the password field before the request resolves, the older successful request can clear the newer value and show success feedback for stale input.
- **Scope:** Track the submitted password snapshot in `app/dashboard/settings/page.tsx`; only clear the field and show success if the visible value still matches the submitted value. Add a focused regression test in `__tests__/settings-password-feedback.test.tsx`.
- **Expected impact:** Prevents confusing account-management feedback and protects users from losing a newer in-progress password edit.
- **Risk (1-10):** 2
- **Effort/size:** S

## Medium (M): Make password/profile settings controls fully labelled and pending-safe

- **Problem:** The settings form uses visible labels without explicit `htmlFor`/`id` wiring and leaves related fields editable while async account actions are in flight.
- **Scope:** Add explicit label associations, pending disabled states where appropriate, and extend tests for accessible-name queries and pending interactions across profile and password sections.
- **Expected impact:** Improves accessibility and reduces race/confusion across account settings.
- **Risk (1-10):** 4
- **Effort/size:** M

## Large (L): Account security activity and recovery surface

- **Problem:** Users have minimal visibility into account security events and no richer recovery guidance after password/profile failures.
- **Scope:** Add an account-security panel with recent auth events if supported by Supabase, clearer recovery CTAs, and localized explanatory copy.
- **Expected impact:** Stronger trust and user confidence for compliance-sensitive accounts.
