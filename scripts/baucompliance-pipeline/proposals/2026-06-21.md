## S — Label the protocol defect input for accessible review

- **Problem:** In the dashboard handover wizard, the defect description textarea is visually introduced by `defect-detected` text but is not programmatically associated with that label. Screen-reader users and RTL tests cannot reliably query or describe the field by its visible purpose.
- **Scope:** Add a stable `id`/`htmlFor` association for the defect textarea and focused regression coverage that advances to Step 2 and queries the textarea by `defect-detected`.
- **Expected impact:** Improves accessibility and testability on the core protocol finalization flow with minimal product risk.
- **Risk (1-10):** 2
- **Effort/size:** S

## M — Lock the dashboard "new protocol" reset during PDF generation

- **Problem:** After protocol generation, the download action can be pending while the adjacent "new protocol" action remains enabled. A user can reset the form while the PDF request is still resolving, risking stale completion feedback or confusing state.
- **Scope:** Disable/guard the new-protocol reset while `isGenerating` is true on the success step and add controlled-promise coverage for a pending download.
- **Expected impact:** Reduces async UI races in the protocol completion workflow.
- **Risk (1-10):** 4
- **Effort/size:** M

## L — Create a reusable form-field accessibility contract for dashboard workflows

- **Problem:** Dashboard pages have accumulated several independently fixed accessibility and async-state patterns. Without shared field/test helpers, similar label and pending-state regressions can recur across cases, vault, settings, and protocol creation.
- **Scope:** Define reusable dashboard form-field components or conventions, migrate multiple surfaces, and add a cross-page accessibility regression suite.
- **Expected impact:** Strategic consistency and lower future regression risk across the product.
- **Risk (1-10):** 7
- **Effort/size:** L
