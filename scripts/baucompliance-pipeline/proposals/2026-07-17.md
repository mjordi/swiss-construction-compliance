## Proposal S — Show linked protocol count in the case action snapshot

- **Target user and job-to-be-done:** Project admins scanning the Cases ledger to decide whether a defect case already has a supporting protocol trail before opening details or creating another protocol.
- **Problem:** Case cards show next action, deadline countdown, reminder readiness, and a small protocol badge/link, but the read-only scan-level action snapshot omits protocol evidence. Users can miss whether a case already has linked protocols when evaluating audit readiness.
- **Scope:** Add a localized linked-protocols cell to the existing `cases-action-snapshot-*` row using already-loaded protocol counts. No persistence, routing, deadline math, or protocol creation behavior changes.
- **Expected impact:** Faster case triage and clearer audit confidence because protocol evidence appears beside next action, deadline, and reminder readiness.
- **User-visible outcome:** Each case action snapshot states the number of linked protocols or that none is linked yet.
- **Competitive / best-product rationale:** Best-in-market compliance tools surface evidence completeness in the same scan context as deadline urgency.
- **Risk:** 2/10.
- **Effort/size:** S.
- **Phase-1 slice:** Protocol count only in the existing case action snapshot; richer chronological activity remains a future Medium/L slice.

## Proposal M — Add a first read-only case activity summary row

- **Target user and job-to-be-done:** Contractors and admins reviewing whether a case has protocol, reminder, Vault, and checklist activity before deciding next action.
- **Problem:** The case ledger still distributes cross-workflow context across chips, links, details, and Vault cards.
- **Scope:** Expand the scan-level action snapshot into a fuller audit summary with protocol count, Vault evidence status, checklist progress, next action, and reminder readiness.
- **Expected impact:** Meaningful step toward a legal cockpit that reduces navigation and audit uncertainty.
- **User-visible outcome:** Case rows communicate legal artifacts as one audit unit.
- **Competitive / best-product rationale:** Market-leading compliance products make the current evidence trail obvious without forcing users to navigate across modules.
- **Risk:** 5/10.
- **Effort/size:** M.
- **Phase-1 slice:** Today's S proposal adds protocol context; later runs can add Vault evidence status and activity ordering.

## Proposal L — Build a unified audit-ready activity timeline

- **Target user and job-to-be-done:** Owners and legal/admin teams preparing evidence for a defect dispute, warranty review, or deadline audit.
- **Problem:** Cases, protocols, Vault evidence, reminders, and checklist changes are still separate surfaces; users cannot yet see one chronological compliance narrative.
- **Scope:** Define and render a derived timeline model across case, protocol, Vault, reminder, and checklist events.
- **Expected impact:** Major strategic differentiation and legal trust.
- **User-visible outcome:** Every case has a unified audit history that explains what happened and what remains next.
- **Competitive / best-product rationale:** A legal-compliance product should produce evidence narratives, not just store artifacts.
- **Risk:** 8/10.
- **Effort/size:** L.
- **Phase-1 slice:** Define a read-only derived timeline helper for one case surface after the scan-level artifact states are stable.
