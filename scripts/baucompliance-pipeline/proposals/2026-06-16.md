- The Cases page still initializes `const supabase = getSupabase()` directly in render, unlike the Vault page which memoizes it. In unconfigured/local build contexts `getSupabase()` returns a new placeholder client, making effects that depend on `supabase` unstable and risking repeated reloads in tests or development.

## Small (S): Stabilize the Cases page Supabase client

- **Problem:** `app/dashboard/cases/page.tsx` calls `getSupabase()` during every render. When Supabase env vars are absent, the placeholder client is not cached, so the `runCasesRefresh`/`triggerCasesRefresh` dependency chain can become unstable and cause repeated fetch cycles or noisy tests.
- **Scope:** Memoize the Cases page Supabase client with `useMemo(() => getSupabase(), [])` and add a focused regression test that rerenders the page while `getSupabase` would otherwise return a fresh client, proving the initial load is not retriggered by the rerender.
- **Expected impact:** More reliable dashboard cases loading in local/demo/build-test contexts and better consistency with the existing Vault implementation pattern.
- **Risk (1-10):** 2
- **Effort/size:** S

## Medium (M): Normalize Supabase placeholder client identity globally

- **Problem:** The fallback Supabase placeholder returned when env vars are missing is created afresh on each `getSupabase()` call, which can surprise any client component that depends on object identity.
- **Scope:** Cache the placeholder client in `lib/supabase.ts`, audit dashboard pages for render-time client creation, and add tests around missing-env behavior across multiple pages.
- **Expected impact:** Stronger app-wide stability in unconfigured environments and less need for every consumer to memoize defensively.
- **Risk (1-10):** 4
- **Effort/size:** M

## Large (L): Add a unified dashboard data-client provider

- **Problem:** Dashboard pages instantiate and manage Supabase clients independently, making loading/error conventions, retries, and fallback behavior uneven.
- **Scope:** Introduce a dashboard-level data-provider hook with stable client identity, shared error handling, and migration of cases, vault, settings, and dashboard protocol surfaces.
- **Expected impact:** More consistent reliability and simpler future dashboard feature work.
- **Risk (1-10):** 7
- **Effort/size:** L / strategic
