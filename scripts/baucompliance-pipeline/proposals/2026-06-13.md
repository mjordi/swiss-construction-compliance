## Small (S): Wire login form labels and auth feedback live regions

- **Problem:** The login/register form renders visible labels and auth feedback, but the name/email/password labels are not explicitly associated with inputs and success/error messages are not announced as live regions.
- **Scope:** Add stable `id`/`htmlFor` wiring for login fields, mark error/success feedback with `role="alert"` / `role="status"`, and add a focused accessibility regression test.
- **Expected impact:** Improves keyboard and screen-reader reliability on the critical entry path while matching the recent accessibility-improvement cadence.
- **Risk:** 2/10 — mostly markup and test coverage on a single page.
- **Effort/size:** S.

## Medium (M): Keep inline case-edit cache fresh after successful updates

- **Problem:** `app/dashboard/cases/page.tsx` preserves the last successful case list so refresh failures after initial load do not blank the UI. Successful deletes already update that cache, but successful inline edits only update visible `dbCases`; if the follow-up refresh fails or error recovery falls back to the cached list, the edited case can revert to stale values.
- **Scope:** Update `lastSuccessfulCasesRef.current` after a successful inline edit using the same payload applied to `dbCases`, and add a regression test that performs a successful edit, forces the follow-up refresh to fail, and verifies the edited values remain visible with no load error.
- **Expected impact:** Prevents confusing post-save reverts in the case management workflow and strengthens an existing optimistic-state pattern.
- **Risk:** 3/10 — localized to cases optimistic state/cache logic with an existing analogous delete regression test.
- **Effort/size:** M.

## Large / strategic (L): Add a safe client-storage helper for draft persistence

- **Problem:** Several client-storage writes/removes for dashboard drafts, calculator drafts, language preference, and marketing attribution are unguarded. In private or restricted browsers, `localStorage` failures can interrupt compliance workflows even though reads are often already protected.
- **Scope:** Introduce a small SSR-safe storage helper, migrate draft persistence paths first, then add tests that mock throwing storage calls; expand to language/attribution in later phases if needed.
- **Expected impact:** Improves resilience for users in restricted enterprise browsers and reduces a class of client-only crashes.
- **Risk:** 6/10 — cross-cutting persistence behavior and broader test matrix make it too large for one daily PR unless phased carefully.
- **Effort/size:** L.
