## Proposal S — Confirm Vault archive and restore actions inline

- **Target user and job-to-be-done:** Project admins and contractors curating evidence projects in the Technical Vault, deciding whether a case’s proof package should remain active or move into later audit review.
- **Problem:** Archive/restore currently mutates the card immediately, but the user gets no explicit success confirmation after the persistence step. In a legal evidence workflow, a silent state transition can feel risky: did the archive actually persist, or did the card only move optimistically?
- **Scope:** Add per-project localized success feedback after a Vault project archive/restore mutation succeeds; clear stale feedback when the same project is mutated again; keep existing rollback/error behavior. Add focused coverage in the Vault test file.
- **Expected impact:** Improves trust around a legally meaningful evidence lifecycle transition without changing persistence schema, timeline math, or cross-page routing.
- **User-visible outcome:** After archiving or restoring a Vault project, the user sees a concise confirmation such as “Project archived” / “Project restored.”
- **Competitive / best-product rationale:** Audit-ready products should acknowledge legal-state transitions, especially when moving evidence between active and archived review modes.
- **Risk (1-10):** 2
- **Effort/size:** S
- **Phase-1 slice:** Add inline per-card success messages only; defer full audit activity logging.

## Proposal M — Add a first Vault archive activity summary from existing case/protocol state

- **Target user and job-to-be-done:** Team leads reviewing whether a Vault project is active, archived, or needs a follow-up protocol/case action.
- **Problem:** Vault cards now show next legal action and deadline context, but they do not yet summarize the recent evidence lifecycle events that led to the current state.
- **Scope:** Add a read-only “latest activity” line per Vault card derived from existing case `updated_at`, archive status, protocol count, and checklist progress; no new persistence.
- **Expected impact:** Moves toward a unified audit cockpit by making evidence state changes explainable at a glance.
- **User-visible outcome:** Vault cards would show a compact latest-activity summary such as “2 protocols linked · checklist 3/4 · updated today.”
- **Competitive / best-product rationale:** Best products help users understand not only the current status but why it is trustworthy.
- **Risk (1-10):** 5
- **Effort/size:** M
- **Phase-1 slice:** Derive a single localized latest-activity line for active project cards only.

## Proposal L — Build a unified audit-ready activity timeline across cases, protocols, vault evidence, and reminders

- **Target user and job-to-be-done:** Owners and legal/compliance reviewers preparing a complete construction defect audit trail.
- **Problem:** Evidence, protocols, reminders, checklist completions, and deadlines are still distributed across surfaces. Users must mentally assemble the legal history.
- **Scope:** Create a shared timeline model and UI that combines case creation/update, protocol finalization, reminder exports, Vault archive/restore, and checklist completion events.
- **Expected impact:** High strategic value: transforms BauCompliance from separate tools into a legal command center.
- **User-visible outcome:** A case or Vault project exposes an audit-ready chronological activity trail.
- **Competitive / best-product rationale:** A unified legal activity trail is a defensible best-in-market compliance differentiator.
- **Risk (1-10):** 8
- **Effort/size:** L
- **Phase-1 slice:** Start with a read-only derived timeline for one case/Vault card using existing local state; defer database event persistence.
