# Daily Product Improvement Proposals — 2026-07-21

## PM insights

Primary target users for today are Swiss construction owners, contractors, and project admins scanning the Cases ledger under time pressure. Their job-to-be-done is to decide which defect cases are legally audit-ready for a notice package and which cases still need evidence, notices, reminders, or protocol support before a deadline is missed.

Recent repo/product context shows a deliberate 14-day pattern of S-sized but legally meaningful slices: Vault audit snapshots and archive/restore confirmations, protocol/case handoffs, linked-case checklist normalization, and scan-level Cases readiness signals for reminders, linked protocols, evidence, and notice drafts. The repeated strategic backlog is a unified audit/activity cockpit; the strongest opportunity today is to turn the now-complete scan-level ingredients into a first readable audit summary rather than adding another isolated chip.

Best-in-market Swiss construction-compliance software should make legal package readiness obvious at scan time: not only individual fields, but a concise answer to “is this case ready to act on, and what is missing?”

## Proposal S — Add an explicit defect-documentation readiness cell to the case snapshot

- **Target user and JTBD:** Project admins scanning defect cases who need to know whether the basic defect record is documented before notice work proceeds.
- **Problem:** The action snapshot exposes deadline, protocols, evidence, notice, and reminders, but defect documentation itself is only implicit in checklist progress/detail state.
- **Scope:** Add one localized read-only `Defect documented`/`Defect missing` cell to `cases-action-snapshot-*` from normalized `checklist.defectDocumented`.
- **Expected impact:** Completes one missing checklist dimension at scan level.
- **User-visible outcome:** Users can see whether the base defect record is complete before opening details.
- **Competitive / best-product rationale:** Audit-oriented tools should expose every legal readiness prerequisite at scan level.
- **Risk:** 1/10.
- **Effort/size:** S.
- **Phase-1 slice:** The single read-only cell plus focused test coverage.

## Proposal M — Add a first case audit-readiness summary row

- **Target user and JTBD:** Construction owners, contractors, and admins triaging the Cases ledger who need a quick answer: “is this case ready for legal follow-up, and what still blocks it?”
- **Problem:** Recent work surfaced individual readiness dimensions, but users still have to mentally combine checklist progress, reminder status, linked protocols, evidence, and notice state.
- **Scope:** Add a compact localized read-only summary at the top of each case action snapshot that counts ready audit-package dimensions and names missing items using the already-normalized checklist state plus existing linked protocol counts.
- **Expected impact:** Turns several prior S-sized prerequisites into a more useful phase-1 audit cockpit signal without changing persistence or workflows.
- **User-visible outcome:** A user scanning a card sees a concise `ready count + missing items` summary before reading the detailed cells.
- **Competitive / best-product rationale:** A best-in-market compliance cockpit should summarize legal package readiness, not force users to synthesize it from isolated chips.
- **Risk:** 3/10.
- **Effort/size:** M, bounded frontend/read-only slice.
- **Phase-1 slice:** Summary row only; no new routing, persistence, deadline math, checklist schema, filtering, or full timeline model.

## Proposal L — Build a unified audit-ready activity timeline across cases, protocols, Vault evidence, and reminders

- **Target user and JTBD:** Owners/contractors preparing for legal review who need a chronological, defensible activity record across all compliance artifacts.
- **Problem:** Case, protocol, Vault, evidence, and reminder signals remain distributed across multiple screens.
- **Scope:** Create a cross-workflow derived timeline model, surface it on Cases and Vault, and connect it to protocol/evidence/reminder events.
- **Expected impact:** Step-change toward an audit cockpit and stronger market positioning.
- **User-visible outcome:** Users can review the full compliance trail from defect discovery through notice package readiness.
- **Competitive / best-product rationale:** Leading compliance products should provide an audit-ready chronology, not just task lists.
- **Risk:** 8/10.
- **Effort/size:** L.
- **Phase-1 slice:** Start with a derived read-only timeline helper for one case card after the scan-level summary contract is validated.
