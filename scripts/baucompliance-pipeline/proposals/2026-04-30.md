# Research Proposals — 2026-04-30

## PM Insights
- The last week of merged work is concentrated on narrow dashboard-wizard hardening, mostly small UX/data-integrity patches that land cleanly in 1–6 files.
- The product’s strongest differentiator remains legally defensible workflow records, so draft/state bugs that can reintroduce stale protocol data are higher value than cosmetic polish.
- Existing code already persists wizard drafts locally, but successful finalization does not clear the persisted draft, which risks restoring stale handover details into the next protocol session.

## Proposal S — Clear persisted wizard draft after successful protocol finalization
- **Title**: Clear persisted wizard draft after successful protocol finalization
- **Problem**: After a protocol is finalized, the local draft remains in browser storage. Returning to the wizard can restore stale project/client/defect data, increasing the risk of accidental reuse in a new protocol.
- **Scope**: Remove the persisted local draft immediately after successful finalization while preserving the in-memory success screen for download/new-protocol actions.
- **Expected impact**: Reduces stale-data carryover and strengthens record hygiene for repeated protocol creation.
- **Risk (1-10)**: 2
- **Effort/size**: Small

## Proposal M — Add finalize-readiness helper and disable finalize until defect acknowledgment is complete
- **Title**: Add finalize-readiness helper and disable finalize until defect acknowledgment is complete
- **Problem**: The finalize CTA already blocks missing signatures, but users only learn about missing defect acknowledgment after clicking finalize, creating avoidable friction at the most sensitive step.
- **Scope**: Add a small readiness helper in step 2 and disable finalize until either a defect description is entered or “no visible defects” is confirmed.
- **Expected impact**: Fewer failed finalize attempts and clearer completion guidance.
- **Risk (1-10)**: 3
- **Effort/size**: Medium

## Proposal L — Add authenticated cross-device protocol draft sync
- **Title**: Add authenticated cross-device protocol draft sync
- **Problem**: Draft continuity is still local-only, so field users switching devices can lose progress and recreate records manually.
- **Scope**: Introduce server-backed draft persistence, hydration, conflict strategy, and recovery UX for the protocol wizard.
- **Expected impact**: Significant workflow resilience for teams moving between field and office devices.
- **Risk (1-10)**: 8
- **Effort/size**: Large
