## S — Lock case row actions while checklist persistence is pending

- **Problem:** When a follow-up checklist checkbox is saving, the checklist inputs and reminder export are disabled, but the same case's edit and delete controls remain active. A user can start deleting or editing a case while an optimistic checklist write is unresolved, risking confusing UI feedback and stale persistence on the same row.
- **Scope:** Treat `checklistSavingByCase[item.id]` as a row-level pending mutation for edit/delete controls on that case, and add focused regression coverage that those sibling actions are disabled while the checklist save is unresolved and re-enabled after success.
- **Expected impact:** Improves reliability on the legal follow-up workflow by preventing conflicting mutations on the same compliance case.
- **Risk (1-10):** 2
- **Effort/size:** S

## M — Add resilient retry controls for stale cached cases refreshes

- **Problem:** The cases page preserves the last successful cases/protocol counts after refresh failures, but follow-up controls can still look current without an explicit retry affordance when a background refresh fails after initial load.
- **Scope:** Add non-blocking stale-data feedback and a retry action for background refresh failures while keeping last-known-good rows visible.
- **Expected impact:** Helps users recover from transient Supabase/network failures without losing context.
- **Risk (1-10):** 4
- **Effort/size:** M
