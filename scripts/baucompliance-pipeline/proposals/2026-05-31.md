## Proposal S — Preserve settings edits made while profile save is in flight

- **Problem:** On the settings page, `handleSaveProfile` submits a normalized profile snapshot, then on success writes that submitted snapshot back into the form fields. If the user edits name/company while the save request is pending, the late success can overwrite the newer local edits and show a misleading saved state.
- **Scope:** Add a focused guard around profile-save completion so the saved snapshot updates, but visible form fields and success feedback are only reset when the fields still match the submitted values. Add a component regression test with a controlled pending Supabase update.
- **Expected impact:** Prevents accidental loss of in-progress profile edits and improves trust in account settings feedback.
- **Risk:** 3/10 — localized to settings profile state; main risk is test mock complexity around Supabase and auth.
- **Effort/size:** S

## Proposal M — Add accessible settings form labels and pending-state lockout

- **Problem:** Settings profile/password inputs rely on visual labels without explicit `htmlFor`/`id` wiring, and pending mutations are only partially communicated to assistive tech.
- **Scope:** Give profile and password controls explicit label associations, expose pending status accessibly, and add tests that query by label instead of placeholder text.
- **Expected impact:** Better accessibility and more resilient tests for a core account-management surface.
- **Risk:** 4/10 — low code risk, but touches multiple settings controls and existing tests.
- **Effort/size:** M

## Proposal L — Account activity and security event timeline

- **Problem:** Contractors have little visibility into account-sensitive events such as password changes, profile updates, or sign-outs, even though the product handles legal/compliance evidence workflows.

- **Expected impact:** Strong strategic trust/compliance story for construction firms managing legal records and evidence.
