# Daily Product Improvement Proposals — 2026-07-16

## PM insights

Primary user lens: Swiss construction owners, contractors, and project admins who scan defect cases to decide which legal deadline or evidence action needs attention next. The workflow moment is the case-list triage pass: users should not have to expand every case to know whether a deadline-protection calendar artifact is ready.

Best-in-market expectation: a Swiss construction-compliance product should make legal readiness visible at scan time, connect cases/protocols/vault/reminders without cognitive load, and keep audit-critical artifact state clear before deadlines are missed.

Recent portfolio signal: the last 14 days are mostly user-facing S-sized slices around reminders, protocol/case handoffs, Vault audit context, and export confidence. The recurring strategic direction is an audit-ready cockpit; the safest next slice is another visible scan-level readiness signal in Cases, not overlapping the open Vault archive-feedback PR.

## Proposal S — Surface calendar-reminder readiness on case cards

- **Target user and job-to-be-done:** Project admins scanning the Cases ledger to decide which defect cases still need deadline protection.
- **Problem:** Calendar reminder readiness exists in the expanded detail panel, but the collapsed case card only shows overall checklist progress. Users can miss that a case still lacks an exported reminder until they expand each row.
- **Scope:** Add a compact localized calendar readiness badge to the case-card header using the existing reminder-readiness contract; no deadline math, persistence, or export behavior changes.
- **Expected impact:** Faster triage and clearer legal confidence for reminder-critical cases.
- **User-visible outcome:** Each case card shows whether calendar export is ready, pending, or not required at scan time.
- **Competitive / best-product rationale:** Best-in-market compliance tools surface audit-critical readiness before users dig into details.
- **Risk:** 2/10.
- **Effort/size:** S.
- **Phase-1 slice:** The header badge only; richer audit cockpit summaries remain backlogged.

## Proposal M — Add a read-only case activity snapshot row

- **Target user and job-to-be-done:** Contractors reviewing whether a project has recent protocol, reminder, vault, and checklist activity before deciding next action.
- **Problem:** Cases show legal timeline details, but cross-workflow activity remains distributed across protocol counts, Vault links, and checklist details.
- **Scope:** Add a compact read-only activity snapshot that combines protocol count, Vault handoff, next action, and reminder state in one scan row.
- **Expected impact:** Stronger step toward the unified audit cockpit.
- **User-visible outcome:** Case rows communicate legal context and artifacts as one audit unit.
- **Competitive / best-product rationale:** Market-leading compliance products make the current evidence trail obvious without forcing navigation.
- **Risk:** 5/10.
- **Effort/size:** M.
- **Phase-1 slice:** Start with reminder-readiness badge and protocol-count/link clarity, then expand in later runs.

## Proposal L — Build a unified audit-ready activity timeline

- **Target user and job-to-be-done:** Owners and legal/admin teams preparing evidence for a defect dispute or deadline review.
- **Problem:** Cases, protocols, Vault evidence, reminders, and checklist changes are still separate surfaces; users cannot yet see a chronological audit trail.
- **Scope:** Design and implement a derived timeline model across case, protocol, Vault, and reminder events, then render it on dashboard and Vault surfaces.
- **Expected impact:** Major strategic differentiation and legal trust.
- **User-visible outcome:** A single audit-ready history for every construction-compliance case.
- **Competitive / best-product rationale:** A legal-compliance product should produce evidence narratives, not just store individual artifacts.
- **Risk:** 8/10.
- **Effort/size:** L.
- **Phase-1 slice:** Define a read-only derived activity helper for one case surface after scan-level readiness signals are complete.
