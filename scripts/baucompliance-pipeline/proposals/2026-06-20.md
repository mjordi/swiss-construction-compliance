## Proposal S — Suppress duplicate login/signup/demo submissions synchronously

- **Problem:** The login page sets `isLoading` after submit starts, but rapid double-submit/click paths can enter the async auth handlers before React commits the disabled state. That can send duplicate login, signup, or demo-login requests and produce confusing feedback.
- **Scope:** Add a synchronous in-flight ref guard shared by login, signup, and demo-login submit paths; clear it on validation/auth failure and signup success while intentionally keeping it set during successful login navigation. Add focused Vitest coverage for duplicate submit suppression.
- **Expected impact:** Prevents accidental duplicate auth requests and aligns the auth surface with the repo's recent pending-mutation reliability standards.
- **Risk:** 2/10 — constrained to `app/login/page.tsx` and one existing login test file.
- **Effort/size:** S

## Proposal M — Add stale-completion protection to login/signup feedback

- **Problem:** If a user changes login mode or edits credentials while an auth request is pending, late async failures/successes can update feedback for an older request.
- **Scope:** Track request tokens and submitted snapshots for auth actions, suppress stale feedback, and add tests for mode toggles/input edits before promise resolution.
- **Expected impact:** Stronger auth UX under poor network conditions and fewer confusing messages after user intent changes.
- **Risk:** 5/10 — touches more auth state transitions and requires careful success-navigation behavior.
- **Effort/size:** M

## Proposal L — Centralize async form mutation guards across dashboard workflows

- **Problem:** Multiple pages implement their own pending-state, duplicate-submit, and stale-completion protections, increasing the chance of inconsistent behavior.
- **Scope:** Introduce a small shared hook/utility for guarded async actions, migrate login plus one dashboard mutation surface, and document usage patterns.
- **Expected impact:** Strategic reduction in race-condition regressions across the product.
- **Risk:** 7/10 — broader refactor across independently-tested workflows; too large for a single daily PR unless phased carefully.
- **Effort/size:** L
