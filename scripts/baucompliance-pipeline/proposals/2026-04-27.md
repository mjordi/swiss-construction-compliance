# Research Proposals — 2026-04-27

## PM Insights (market + repo signal)
1. **Delivery signal:** Last-7-day merged work remains S/M-only, which supports shipping reliability fixes with tight scope rather than platform-scale rewrites.
2. **Workflow signal:** The wizard still allows a restored linked-case ID to go stale if the case is later removed, creating avoidable confusion and linkage risk in a core flow.
3. **Compliance signal:** Defensibility requirements are growing from “document captured” to “event trace available,” which keeps protocol lifecycle auditing as a strong strategic bet but not a same-day item.

## Proposal S — Stale linked-case recovery banner in wizard
- **Size**: Small
- **Problem**: A restored `selectedCaseId` can reference a case that no longer exists for the user.
- **User value**: Prevents confusing broken linkage and keeps protocol finalization safe.
- **Implementation sketch**: Detect stale linked case after case list loads, show localized warning with one-click unlink, and avoid persisting stale linkage on finalize.
- **Risk**: Low

## Proposal M — Finalize progress lock + duplicate-submit guard
- **Size**: Medium
- **Problem**: Users can click finalize repeatedly during slow network/save moments, increasing duplicate-write risk.
- **User value**: More predictable finalization and reduced accidental duplicate protocols.
- **Implementation sketch**: Add explicit in-flight lock, disable destructive controls during save, and show localized “saving protocol” progress state.
- **Risk**: Medium-Low

## Proposal L — Protocol lifecycle event timeline (phase 1)
- **Size**: Large
- **Problem**: Users cannot inspect a structured history of protocol lifecycle actions.
- **User value**: Stronger compliance defensibility and easier dispute preparation.
- **Implementation sketch**: Add append-only protocol events table, write events on key transitions, and expose read-only timeline in case detail.
- **Risk**: High
