## Proposal S — Associate Ruegefrist calculator visible date labels with inputs

- **Target user and job-to-be-done:** Public prospects and occasional dashboard users checking a Swiss defect-notice deadline with keyboard/screen-reader support.
- **Problem:** The public calculator date inputs use `aria-label` but the visible labels are not programmatically associated with the inputs via `htmlFor`/`id`, making the form less robust for assistive technology and automated support checks.
- **Scope:** Add stable ids to the contract/discovery date inputs in `components/ruegefrist-calculator.tsx`, associate the existing visible labels with `htmlFor`, and add focused RTL coverage using label queries.
- **Expected impact:** Improves accessibility and support confidence for a conversion-critical public calculator.
- **User-visible outcome:** Users can focus and identify the date fields through their visible labels.
- **Competitive / best-product rationale:** Legal calculators should be accessible and unambiguous, especially for public trust and procurement accessibility reviews.
- **Risk:** 2/10.
- **Effort/size:** S.
- **Phase-1 slice:** Complete as one small accessibility/test patch.

## Proposal M — Canonicalize invalid vault tab query params while preserving project search

- **Target user and job-to-be-done:** Project teams and support users opening shared evidence-vault links to find a project, archived project, or related case context.
- **Problem:** `parseVaultTab` already defaults invalid `tab` values to `projects`, but the URL is not cleaned. A link such as `/dashboard/vault?tab=stale&q=Alpine` renders the default projects view while leaving a misleading invalid tab in the shareable URL.
- **Scope:** In `app/dashboard/vault/page.tsx`, normalize owned vault params when search params are consumed: delete invalid `tab` values while preserving valid `q` and unrelated params. Add focused tests in `__tests__/vault-url-sync.test.tsx` for initial invalid cleanup and external rerender cleanup where parsed tab state is already unchanged.
- **Expected impact:** Makes shared vault links canonical and trustworthy without touching persistence or broader navigation.
- **User-visible outcome:** Stale/invalid vault tab links are cleaned to the actual visible vault state while keeping the user’s project search.
- **Competitive / best-product rationale:** Evidence repositories in compliance workflows should not leave ambiguous state in URLs that may be used in support, audit, or contractor handoff conversations.
- **Risk:** 3/10.
- **Effort/size:** M (small code change but medium product value as a cross-workflow URL-state trust slice).
- **Phase-1 slice:** Normalize only the existing `tab` parameter contract; do not add new filters or vault IA.

## Proposal L — Case activity timeline across protocols, vault, checklist, and reminders

- **Target user and job-to-be-done:** Project leads preparing a defensible audit trail for a defect, from discovery through notice, evidence, protocol finalization, and reminder export.
- **Problem:** Case, vault, protocol, and reminder events are split across surfaces. Users must infer whether a case has a complete legal action trail.
- **Scope:** Create a unified case activity timeline model and eventually render it on case detail/card surfaces.
- **Expected impact:** High legal trust and differentiation: users can see a coherent compliance trail rather than isolated actions.
- **User-visible outcome:** A case shows when defects were documented, evidence attached, reminders exported, and protocols generated.
- **Competitive / best-product rationale:** Best-in-market compliance software makes the audit trail obvious and exportable.
- **Risk:** 7/10.
- **Effort/size:** L.
- **Phase-1 slice:** Start with a pure `buildCaseActivityTimeline` helper and fixture tests only; defer UI/persistence until a later PR.
