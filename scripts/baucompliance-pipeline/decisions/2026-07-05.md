## Decisions

### APPROVED — Label vault compliance readiness with completed checklist counts

**User-impact note:** Target users are Swiss construction owners, contractors, and project admins scanning the Technical Vault to decide whether project evidence is audit-ready, needs follow-up, or can be archived. The workflow moment is card-level triage before opening a case or archiving/restoring a vault project. This is not just incremental polish: the existing readiness percentage is a legal-confidence signal, and showing the completed/total checklist count makes that signal explainable at the decision point.

**Engineering task:** In `app/dashboard/vault/page.tsx`, retain the existing checklist progress for each vault project card, render a localized compact count label beside the compliance percentage, and add focused coverage in `__tests__/vault-follow-up-link.test.tsx` proving the count reflects default/persisted checklist state.

### BACKLOG — Add a vault evidence-readiness summary by missing legal step

Valuable next step, but broader than today because it needs domain copy for missing evidence/notice/reminder steps across locales and more review of next-step priority. The approved count label is a safe phase-1 explanation of the same readiness contract.

### BACKLOG — Unified compliance activity cockpit across cases, vault, protocols, and reminders

Strategically strongest, but too large for one daily PR. The next credible phase is to keep enriching shared readiness/action contracts in existing surfaces before introducing a new cockpit route.
