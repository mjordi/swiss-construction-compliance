## Decisions

### APPROVED — Canonicalize invalid cases filter query params

**User-impact note:** Target users are project teams and support users sharing filtered cases dashboard links during defect triage. The workflow moment is opening or sharing a cases URL with legal-regime/status/sort/search filters. This is not just incremental polish: invalid legal-triage filters in a URL can make a shared view look narrower or more legally specific than the UI actually applied, weakening trust in audit/support conversations.

**Engineering task:** In `app/dashboard/cases/page.tsx`, normalize invalid `regime`, `status`, and `sort` query params whenever cases search params are consumed, preserving valid sibling params and unrelated params. Add focused regression tests in `__tests__/cases-url-sync.test.tsx` proving invalid params are removed on initial load and on external query-param changes where the parsed filter state is otherwise unchanged.

### BACKLOG — Associate Ruegefrist calculator visible date labels with inputs

Valuable and very safe, but it is another small accessibility consistency slice. Backlog it because today has a stronger cases share-link trust opportunity with higher workflow impact.

### BACKLOG — Case activity timeline across protocols, vault, and reminders

Strategically compelling, but still too large for one autonomous daily PR. The next credible phase-1 slice should be a pure timeline derivation helper with fixture tests before UI or persistence.
