## Decisions

### APPROVED — Explain missing linked-case handoffs in the protocol engine

**User-impact note:** Target users are Swiss construction owners, contractors, and project admins who open the dashboard protocol engine from a case/vault handoff expecting linked legal context to prefill and guide the protocol. The workflow moment is the handoff boundary before starting the protocol. This is not just incremental polish: when a legal case context cannot be applied, silently falling back to a standalone form can produce an incorrect or untrusted protocol workflow; the product should explicitly show that the linked case was not found.

**Engineering task:** In `app/dashboard/page.tsx`, track an unresolved requested `case=` handoff after cases load successfully, show the existing `dashboard-linked-case-missing-*` warning for that missing handoff, clear it when dismissed or when a valid case is selected, and add focused coverage in `__tests__/dashboard-linked-case-retry.test.tsx`.

### BACKLOG — Add a protocol handoff context summary before finalization

Valuable Medium follow-up because users should verify linked case/deadline context right before PDF/export. It is broader than today because it needs finalization-layout review and should ideally include the selected case context in the exported artifact strategy.

### BACKLOG — Unified compliance activity cockpit across cases, vault, protocols, and reminders

Strategically strongest, but too large for one daily PR. The smallest credible phase-1 is a read-only activity card fed by existing case timeline and vault readiness contracts after the handoff/finalization confidence surfaces are more complete.
